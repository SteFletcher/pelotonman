02 · System Architecture

Software Architecture Description (SAD)

1. Architectural Style

Pelotonman is a layered, engine-first single-page application with a strict separation between pure game logic and imperative UI. The design rests on three principles, identical in spirit to its football sibling:

  1. Engine purity. Every module under src/engine/ is a pure ES module with no DOM access and no framework. It runs identically in the browser and in Node, which is what makes the test suite possible without a DOM shim.
  2. Single source of randomness. One seeded mulberry32 RNG instance threads through the entire career. Determinism is a first-class architectural property, not an afterthought.
  3. UI as a thin client. js/app.js only renders state produced by the engine and forwards user gestures back to engine methods. It owns no game rules.
Key difference from champman

Champman generated fictional squads deterministically from a club-name hash. Pelotonman ships real, hand-authored rosters for all 24 teams (real rider names, real roles). The data module is therefore larger and is treated as a versioned data asset rather than a generator. Determinism still holds for everything after the initial world load — race outcomes, AI behaviour, development, transfers — because that is where the RNG is consumed.

4. Dependency Rules

  • The dependency graph is a DAG with rng.js as the root. No cycles.
  • game.js depends on every engine module; it is the only module that composes the subsystems.
  • race.js depends only on rng.js and squad.js — it knows nothing about seasons, finances, or transfers.
  • gt.js depends on race.js (a Grand Tour is a sequence of races) and series.js (for jersey points accumulation).
  • app.js imports Game and a few helpers (teamRatings, overallRating, ability) for display; it never imports race.js directly.
  • The data module teams.js depends on riders.js only (to build rider records from the authored rosters).

5. The Determinism Contract

Determinism is enforced by three rules that every engine author must obey:

Rule 1 — One RNG

All randomness flows through a createRng(seed) instance. Math.random() appears only to generate a seed when none is supplied. The career's RNG lives on game.rng and its state is serialized with the save.

Rule 2 — Serializable state

The RNG exposes getState() / setState(). Game.serialize() captures rngState alongside all team/world state; Game.restore() reconstructs the instance and re-injects the RNG state, so a resumed career draws the same future random numbers as the original.

Rule 3 — No hidden mutation

The race engine never mutates the rider objects it is given; per-race state lives in StatLine objects keyed by rider id. This keeps the squad objects stable across the sim and the UI.

Initial world is deterministic by data, not RNG

Because rosters are authored (real names), the initial world consumes no RNG draws. The first RNG consumption is the first AI/uncertain event after the career starts (e.g. the first simulated race). This makes the world load instant and reproducible.

7. Race Engine Internals

RaceSim is a stateful object exposing playSegment() (returns the events generated that segment), setOrders(), setRacePlan(), and finish(). Internally each segment (typically 5–10 km, finer near key points):

  1. Runs AI race-plan policy (defend GC / chase break / set up sprinter) and AI orders for all teams.
  2. Resolves the peloton pace from the controlling teams' orders and the road profile (climb sector flattens pace).
  3. Evaluates breakaway formation and catch: a break's survival is a function of gap, headwind, terrain, and chase intensity.
  4. Generates quiet domestique work (pulls) that feed the live ratings and burn condition.
  5. Resolves crashes & mechanicals (weighted by injuryProne and dense sections); dropped riders lose time or abandon.
  6. On a climb sector: re-groups by climbing strength, time gaps open, KOM points awarded at the summit.
  7. On the final sector: resolves the sprint (SPR-led) or summit finish (CLM/LDR) and assigns placings + time gaps + points.
  8. Decays condition for every rider who pulled or chased.
  9. Pushes a timeline snapshot and checks for stage/race end.

The strength model is role- and profile-aware: a climber's climb dominates on Cat-1 ascents; a sprinter's power dominates in the final 500 m of a flat stage; a time trialist (high power, high consistency) dominates an ITT. Domestique contributions are pooled per-team and applied as pace modulation, so a full sprint train genuinely delivers a sprinter to the line fresher than a lone sprinter.

8. Persistence Model

Saves are a single JSON blob in localStorage['pelotonman-save']. The shape is a flat snapshot of the Game instance minus the RNG object (whose state is stored separately as rngState).

{
  version: 1,
  managerName, teamName,
  rngState,                 // ← the determinism key
  teams: [...],             // full world state
  freeAgents, seasonIndex, week, calendar,
  results,                  // one-day + stage-race results
  grandTours: { giro, tour, vuelta },   // live GT state
  seriesStandings, seasonStats, monthly, inbox,
  pendingOffers, nextOfferId,
  board, racePlan, history, youthSeq,
  sacked, reputation, jobOffers, pendingJobOffer,
  scouted, pendingScouts
}

Restore constructs a throwaway Game and then Object.assigns the saved fields over it before re-injecting the RNG state. A version mismatch throws, refusing to load incompatible saves.
